# Opportunity Intelligence Engine Architecture

This project is now structured as a production-style autonomous opportunity intelligence pipeline, not a single prompt script.

## 1. System Architecture

Execution flow:

```text
DiscoveryPlanner
  -> bounded ecosystem tasks
  -> OpenCodeRunner with model routing
  -> CheckpointStore for every attempt
  -> JsonExtractor with partial-output salvage
  -> OpportunityRecord Pydantic normalization
  -> ValidationEngine live page checks
  -> NormalizationEngine classification and dedupe
  -> deterministic JSON export
```

Core modules live in `search_hackathons.py`:

- `DiscoveryPlanner`: splits research into small ecosystem-specific tasks.
- `OpenCodeRunner`: runs OpenCode with model fallback, timeout handling, retries, and cache reuse.
- `CheckpointStore`: stores raw attempt output and run summaries under `.opportunity_cache/`.
- `JsonExtractor`: recovers strict JSON, single candidate objects, and partial candidate objects.
- `OpportunityRecord`: Pydantic v2 schema for field normalization and type repair.
- `ValidationEngine`: deterministic URL, lifecycle, deadline, and source validation.
- `NormalizationEngine`: final active, borderline, and excluded classification.

Default model order:

1. `opencode/big-pickle`
2. `opencode/hy3-preview-free`
3. `cloudflare-workers-ai/@cf/moonshotai/kimi-k2.6`

Users can pin one model:

```bash
python3 search_hackathons.py -m hy3
```

## 2. Discovery Engine

The engine no longer sends one huge prompt. It creates bounded discovery tasks:

- Central challenge portals: MyGov, Innovate India, SIH, Data.gov events.
- AI and digital public infrastructure: IndiaAI, AIKosh, Bhashini, MeitY.
- Defence and space: iDEX, DRDO, ISRO.
- Bio, health, science, R&D: BIRAC, DBT, DST, TDB, NHA.
- Startup and MSME: Startup India, DPIIT, MSME.
- State and academic ecosystems: state startup missions, AICTE, `.ac.in` portals.

Each task has:

- seed URLs
- search queries
- a strict candidate cap
- current-date validation rules
- a small output schema

This reduces timeout risk, token load, and reasoning loops. Saturation detection can stop a run after repeated low-novelty tasks:

```bash
python3 search_hackathons.py --stop-on-saturation
```

## 3. Validation Engine

The live validator probes candidate URLs and records:

- final URL after redirects
- HTTP status code
- page accessibility
- open-registration keywords
- hard closed keywords
- form or submit affordances
- auth-wall hints
- stale-cycle and zombie-page signals
- deadline freshness against `--current-date`

Important behavior:

- A trusted official source is required for active inclusion.
- External platforms are allowed only as registration surfaces, not primary evidence.
- Closed navigation text does not automatically reject an active page if current open evidence and future deadlines are present.
- Hard closed phrases such as `registration closed` or `no longer accepting` remain fatal.
- Grant/R&D calls are moved to `borderline_opportunities` when they are not truly competitions.

To validate an existing file without running OpenCode:

```bash
python3 search_hackathons.py \
  --validate-existing hackathons_results.json \
  --current-date 2026-05-08
```

For offline deterministic validation:

```bash
python3 search_hackathons.py \
  --validate-existing hackathons_results.json \
  --skip-live-validation
```

## 4. Normalization Engine

The final JSON is generated by code, not trusted directly from the LLM.

Normalization repairs:

- stringified JSON fields such as `team_size` and `submission_fee`
- URL tracking parameters
- trailing slash URL variants
- enum aliases
- date formats
- boolean-like strings
- missing IDs and slugs
- confidence score bounds

The schema accepts extra fields so future opportunity types can be added without breaking ingestion, while the final export remains deterministic.

## 5. Recovery and Resilience

Failure modes now handled:

- model provider errors
- timeout per model attempt
- partial OpenCode output
- invalid JSON wrappers
- single candidate object instead of full payload
- stale cached data
- crash during final export

Recovery systems:

- every attempt writes raw output and metadata to `.opportunity_cache/runs/<run_id>/`
- successful task outputs are cached by task hash and current date
- final JSON writes atomically through a temp file
- partial candidate objects can still be salvaged from broken JSON
- empty timeout runs produce valid JSON metadata instead of crashing

## 6. Performance Optimization

Performance improvements:

- small task prompts instead of one 3,000+ line prompt
- candidate cap per task
- cache reuse by current date
- model fallback per task instead of entire run failure
- optional `--max-tasks` smoke testing
- optional offline validation
- live validation probes only the highest-value URLs per candidate

Useful commands:

```bash
python3 search_hackathons.py --dry-run --max-tasks 2
python3 search_hackathons.py --max-tasks 1 --timeout 120 -m hy3
python3 search_hackathons.py --refresh-cache
```

## 7. Data Quality

Anti-hallucination controls:

- official source requirement
- current-date deadline check
- lifecycle keyword analysis
- zombie page detection
- URL accessibility check
- LLM confidence is capped and recalibrated
- borderline separation for grants, R&D calls, and uncertain portals
- dedupe by canonical URL and normalized title
- source and validation metadata retained in every record

Confidence is not treated as an LLM claim. It is recalculated after validation evidence is available.

## 8. Future Scalability

The architecture can scale into adjacent ecosystems by adding new `DiscoveryTask` packs and classifier rules, not by rewriting the runner.

Future opportunity families:

- government startup grants
- private incubators
- IIT, IIIT, NIT, and IISc ecosystems
- SDG funding programs
- CSR innovation programs
- accelerator ecosystems
- venture grants
- R&D calls
- incubation opportunities
- fellowships
- global innovation programs

Recommended future extension:

- add an `opportunity_family` enum separate from `event_type`
- add per-family schemas for grants, fellowships, accelerators, and competitions
- keep the shared validation layer for URL, deadline, source, and lifecycle state
- add a persistent database for longitudinal freshness checks
- add Playwright-based JS portal validation for pages requiring client rendering
- add queue-based async crawling when volume grows beyond local CLI execution
